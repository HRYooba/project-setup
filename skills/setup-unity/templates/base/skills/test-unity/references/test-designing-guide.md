# テスト設計ガイド（何をテストするか）

テストケースを「何を・どの粒度で検証するか」を決めるための判断基準。
**コードを書く前（または書き直す前）にこのガイドで設計し、実装規約は `test-writing-guide.md` に従う。**

大原則は 2 つ:

1. **テストは仕様（観測可能な振る舞い）を守る安全網である。** 実装をなぞる写経ではない。
2. **書きすぎない。** AI はテストを量産しがちで、低価値テストは保守コストだけ増やす。
   「このテストだけが殺せる回帰」を 1 文で言えないテストは追加しない。

---

## 1. テスト対象の選別

「原則テストする変更」「テスト不要と判断してよい変更」の一覧は `rules/testing.md`「テスト責任」を
単一ソースとする（呼び出し順序・キャンセル伝播などを含む）。ここでは重複させず、テスト設計時に
補足が要る観点だけを以下に挙げる。

### MonoBehaviour
- MonoBehaviour のランタイム挙動は原則スキップ（手動確認に委譲）。
- MonoBehaviour 内に純ロジックが埋まっている場合は、テストをこじ開けるのでなく plain class への分離を検討する（§9）。

---

## 2. このプロジェクトのテスト層

本プロジェクトの自動テストは **EditMode の単体 / ロジックテストに限定**する。

- 検証するのは「**メソッドの直接呼び出しで起動する振る舞い**」（純ロジック、状態遷移、エラー変換等）。
- PlayMode / 統合テスト / 視覚検証 / 画像解析は**自動スイートでは扱わない**。ランタイム挙動・見た目・マルチプレイヤー挙動は**ユーザー手動確認**に委譲する（`rules/testing.md`「ランタイム動作確認フロー」）。
- したがって設計対象は常に「unit / logic test」であり、層を PlayMode へ分割しない。

---

## 3. テスト技法の選択

### spec-based > structural（最重要原則）
**仕様ベース（観測可能な振る舞いを検証）を、構造ベース（実装の内部分岐をなぞる）より優先する。**
構造テストはリファクタで壊れ、価値が急速に失われる。実装に不安があるとき一時的に構造テストを書くのは可だが、**仕様テストが同じ振る舞いを覆ったら削除する**。private 実装詳細に依存したテストは原則作らない。

### 技法
| 技法 | 使う場面 |
|:-----|:---------|
| 同値分割 | 入力を valid / invalid に分け、各区分の代表 1 件 |
| 境界値分析 | 各区分の境目。過剰にやらずパラメータ化で集約。仕様が境界で挙動を変えないなら代表 1 件で可 |
| 状態遷移 | FSM を持つ対象。1 ケース = 0-switch（A→B の直接遷移）|
| デシジョンテーブル | 複数条件の組合せで結果が変わる場合 |
| エラー推測 | 経験則。キャンセル中の操作、二重呼び出し、数値オーバーフロー、ネットワーク失敗等 |

### 無効同値（invalid partition）の扱い
- **UI から入りうる無効入力**（数値フィールドの範囲外等）= システム境界の実在する失敗経路 → テストする。
- **仕様が挙動を定義していない無効入力**（負値・null 等で何が起きるべきか不明）→ **推測せず `AskUserQuestion` で確認**してから設計する。

---

## 4. 依存のエラー経路をテストするか（外部 vs 自前）

依存の失敗パスを検証すべきかは、その依存の出自で決める:

| 依存の出自 | 判断 |
|:-----------|:-----|
| **外部ライブラリ / framework / SDK**（本プロジェクトの所有外。例: 音声/ネットワーク SDK・HTTP backend） | **テストする。** stub でエラー条件を注入し、エラー変換・ガード・状態整合を検証する |
| **自前コード**（本プロジェクトの別コンポーネント） | **skip。** 内部コードの正しさは信頼し、その依存の単体テスト側で守る |
| **不明** | `AskUserQuestion` でユーザーに確認してから設計する |

> 例: 外部 SDK の未接続時に `NotConnected` へ変換するガードは「外部依存のエラー変換」なので **valuable**。一方、自前の下位コンポーネントが返す失敗を上位でもう一度確認するだけのテストは「自前コードの再検証」なので不要（§6 も参照）。

## 5. テスト価値の判定（追加前ゲート）

追加候補ごとに、**すべて**満たすか確認する。1 つでも満たせないものは追加しない。

### ゲート 1: 回帰特定
そのテストが失敗したとき検出できる**具体的な回帰を 1 文で**説明できる。説明できないなら追加しない。

### ゲート 2: 一意性（反冗長）— 2 軸の重複定義
**真の重複** = 別テストと **condition（setup / 入力）も assertion（観測 / 期待値）も同一**。両方一致して初めて重複。

- condition が違う → 重複でない
- condition 同じ + assertion 違う → 重複でない（別の回帰を殺している）
- **同一 condition のテストを multi-assert に統合しない**（1 ケース 1 検証。ただし状態遷移後に複数プロパティが同時に正しくある必要がある場合のみ、まとめて assert 可）

「このテストだけが殺せる回帰」を言えること。同じガードを公開 API 違いで反復するのは、**入口ごとに別の回帰を殺す場合のみ**許容する（例: Room 送信と DM 送信で別々の SDK 呼び出し入口）。単なる入力値違いの再確認はパラメータ化で 1 メソッドに集約する（§7）。

### ゲート 3: 仕様語 ↔ assertion 整合（反 fake）— 観測可能性
テストの意図・コメントに**仕様語**（順序 / rollback / fallback / エラー変換 / 状態遷移 / キャンセル伝播）が出るなら、それを**実際に観測する assertion** が存在すること。

**検証は「観測可能な性質」だけを述べる。** 以下を検証内容に混ぜない（= fake の温床）:
- なぜこのテストが存在するか / 現在のバグ挙動（rationale）
- テスト機構（属性・sync/async・assertion の形）
- 入力の構築手順

| | 例 | 問題 |
|:--|:--|:--|
| ❌ Bad（rationale 漏れ） | 「同一シードで再現する。RNG が holder に格納されない現状では失敗する（BUG再現）」 | なぜ存在するかを書いている。観測内容のみにする |
| ✅ Good | 「同一シードで 2 回実行するとマップ構造が一致する」 | 観測可能な振る舞いのみ |
| ❌ Bad（順序を主張するが観測しない） | コメント「leave→logout の順序を守る」だが assert は state クリアのみ | 順序を観測していない fake |
| ✅ Good | 呼び出し順を記録し `IndexOf(leave) < IndexOf(logout)` を assert | 仕様語を実観測 |

### ゲート 4: テストダブルの語彙（xUTP）
| ダブル | 役割 | 検証での扱い |
|:-------|:-----|:-------------|
| **Stub** | 定型レスポンスを返し SUT を隔離 | arrange の関心事。検証意図に書かない |
| **Spy** | 呼び出し・引数を記録し後で検証 | **検証対象**。順序・呼び出し回数を観測するなら spy を使う |
| **Fake** | 簡易だが動作する実装 | arrange の関心事。検証意図に書かない |

不要なダブルを増やさない。検証したいのが「呼ばれたか・順序」なら spy、「SUT を動かすため」なら stub/fake。

## 6. 合成クラス（複数コンポーネントを束ねるワークフロー）のテスト方針

下位コンポーネントを合成するクラスのテストは、**下位が単体テストで既に保証している結果 state を再検証しない**。合成クラス固有の契約のみを観測する:

- **呼び出し順序**（leave-before-logout 等）→ 呼び出し順記録（spy / order recorder）で観測
- **継続条件**（下位失敗時に後続を best-effort で続行する 等）
- **キャンセル伝播**（OCE を握りつぶさず後続を実行しない 等）

下位の成果 state を上位で再確認するだけのテストは「実装をなぞるだけ」に該当し追加しない。

---

## 7. 重複検出と統合（書いた後の dedup パス）

テストを書いた後、追加・変更したファイル + 同一テストクラスの既存テストを読み、§5 ゲート 2 の **2 軸定義**で重複を検出する。

- **真の重複**（condition も assertion も同一）→ より正確な名前のものを残し、他を削除。
- **パラメータ化マージ**: condition が違っても「SUT に渡す引数だけが違い、**同一同値区分**で**同一の期待結果**」なら、パラメータ化テストに統合する。
  - パラメータ化テスト本体に `if` / `switch` を書かない
  - **期待値をパラメータ化しない**（期待値が分岐するなら別区分 = 別テスト）
  - 異なる同値区分・異なる期待結果のケースは統合しない

---

## 8. 追加禁止の低品質テスト

追加してはならない低品質テストの一覧は `rules/testing.md`「禁止する低品質テスト」を単一ソースとする
（fake assertion・合成クラスでの結果 state 再検証などを含む）。ここでは重複させない。

**初期値テストの例外**: 外部から購読される状態（State / UI Model 等）の**初期契約**を守る場合に限り許容する。

---

## 9. テスタビリティの評価

設計中に「テストが書きにくい」と感じたら、それは**テストの問題ではなく対象コードの設計問題**の兆候。
無理にテストをこじ開けず、以下のいずれかなら設計見直しを報告する（`rules/coding-standards.md`「設計アプローチ」に従う）:

- 単体テストの境界が引けない（SUT が大きすぎる）
- 状態が隠蔽され外部から観測できない
- 条件数に対しテストケースが O(n) より速く増える（組合せ爆発）
- 依存が注入できない（static / global 結合、constructor 内 `new`）
- MonoBehaviour に純ロジックが埋まっていて分離されていない

この場合、テストを書く前に対象の分割・依存注入化・状態公開を提案する。
